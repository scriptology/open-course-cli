# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Rust terminal (ratatui) AI language tutor. Exercises, curriculum, and answer analysis are generated by an LLM provider the user configures (OpenAI, Anthropic, Google, DeepSeek, Mistral, OpenRouter, Ollama, or a custom OpenAI-compatible endpoint via `rig-core`). Data is stored locally per language pair in LanceDB tables under `.open-course-cli/`.

Read `README.md` for install/usage and `docs/flow.md` for the full user-facing flow (onboarding, dashboard, session lifecycle, settings) — both are kept current and are the source of truth for behavior, not just this file.

## Commands

```bash
cargo run                              # run the TUI (data in ./.open-course-cli/)
cargo run -- --data-dir /path          # run against a different data dir
cargo test                             # run all tests
cargo test --test session_test         # run a single test file
cargo test some_test_name              # run a single test by name
python3 scripts/debug_exercises.py     # debug LLM exercise generation directly
```

Set `OPEN_COURSE_CLI_DEBUG=1` to enable debug event logging (see `src/llm/debug_log.rs`, `src/llm/pipeline.rs::log_debug_event`).

## Architecture

### Event loop and state (`src/app/mod.rs`)

Single-threaded async event loop in `run_app`. One `AppState` struct holds the current `View` enum plus one state struct per view (`DashboardState`, `SessionState`, `CurriculumState`, etc. — all in `src/ui/views/`). Each loop iteration:

1. Draws the current view (`draw()` matches on `state.view`).
2. Kicks off pending background work (e.g. auto-loading session topics/curriculum when a view needs them but has none loaded).
3. `tokio::select!`s over three sources: terminal input (`EventStream`), an `mpsc::Receiver<LlmResult>` for async LLM responses, and a 100ms tick (only redraws when a view is actively waiting on the LLM — `redraw_on_tick`).

Key/mouse handling dispatches to a per-view module (`dashboard::handle_key`, `session::handle_key`, etc. in `src/ui/views/`), which mutate `AppState` directly and return `Result<()>`.

LLM calls are spawned as background tasks that send an `LlmResult` variant (`Exercises`, `Analysis`, `Curriculum`, `TopicReview`, streaming chunks, diagnostics, ...) back through `llm_tx`. `src/app/llm_results.rs::apply_llm_result` matches on the variant and updates the relevant view state — this is the single place async LLM results get applied to `AppState`. When adding a new LLM-backed feature, add a new `LlmResult` variant and handle it there rather than blocking the event loop.

### Views (`src/ui/views/`)

Each view is a module exposing `draw()` and `handle_key()` plus a `*State` struct owned by `AppState`. Views: `onboarding` (multi-step wizard, own `state.rs`/`steps.rs`/`handlers.rs` submodules), `dashboard`, `session`, `docs`, `report`, `curriculum`, `settings` (own `fields.rs`/`data.rs`/`provider_setup.rs` submodules), `model_check`, `pairs`, `update`. Shared widgets (activity calendar, sparkline, progress bars, spinner, markdown rendering) live in `src/ui/widgets/`.

Mouse wheel scrolling is only enabled on views without text input (`view_supports_mouse`); `m` toggles mouse capture off for native text selection.

### Core domain logic (`src/core/`)

- `session/` — session orchestration: `topic_selection.rs` implements `pick_next_session_topic` (balances new curriculum topics vs. spaced-repetition review, roughly 1-in-3 sessions is a review, more when many topics are due), `scoring.rs` (mastery decay ~5%/day), `apply.rs` (applying analysis results to progress/history/learning items), `models.rs` (shared types like `Exercise`, `AnalysisResult`).
- `dashboard.rs` — aggregates progress/history into dashboard stats (weak topics, activity calendar buckets).
- `language.rs` — ISO 639-1 language code handling.

### Persistence (`src/db/`)

One LanceDB-backed table module per concern: `curriculum.rs`, `progress.rs`, `history.rs`, `reviews.rs`, `learning_items.rs`, plus `metadata.rs` and `util.rs` helpers. `Database` (in `src/db/mod.rs`) owns one instance of each table and is cheap to `.clone()` (each table wraps a LanceDB connection handle). Each language pair gets its own `Database` connected to `.open-course-cli/pairs/{native-target}/db/`; switching pairs (`app::switch_pair`) reconnects and replaces `state.db`.

### LLM integration (`src/llm/`)

- `client.rs` / `factory.rs` — `LlmClient` trait and `RigClient` (built on `rig-core`); `create_llm_model` builds a client from the active provider config.
- `provider.rs` — provider enum/metadata (OpenAI, Anthropic, Google, DeepSeek, Mistral, OpenRouter, Ollama, Custom) and per-provider defaults.
- `prompts.rs` — prompt templates for curriculum, exercises, analysis, topic docs.
- `pipeline.rs` — request orchestration and debug logging.
- `streaming.rs` — streamed response handling (used for curriculum generation progress and text streaming).
- `parse.rs` — parsing/validating structured LLM output (exercises, analysis, curriculum JSON) with `schemars`-derived schemas.
- `diagnostics.rs` — the onboarding "model check" pipeline (connectivity, streaming, exercise generation, analysis, topic review), driving `ModelCheckState`.
- `model_listing.rs` — fetching available models per provider for onboarding/settings.
- `retry.rs` / `transport.rs` — retry policy and HTTP transport config.

### Config (`src/config/`)

`OpenCourseConfig` (in `config/mod.rs`) holds provider settings and preferences (global) plus a `profile.rs` per language pair (native/target language, age, CEFR level) and the `active_pair` id. `provider.rs` holds per-provider config (API key, base URL, model, endpoint kind). `migration.rs` handles one-off upgrades of on-disk state between versions (e.g. recreating the curriculum table, migrating from an old markdown-based curriculum format, clearing stale caches) — these run once at startup in `main.rs` and are gated by marker files so they don't re-run.

### Errors

`src/error.rs` defines `AppError`/`Result` used throughout; most fallible functions return `crate::error::Result<T>`.

## Testing conventions

Tests live in `tests/` as integration tests (one file per concern: `config_test.rs`, `db_test.rs`, `dashboard_test.rs`, `session_test.rs`, `settings_layout_test.rs`, `prompts_test.rs`, plus `debug_*`/`layout_debug.rs` exploratory harnesses). They use `tempfile` for isolated data dirs and drive real `AppState`/`Database` instances rather than mocking — prefer the same style for new tests (construct real state, assert on behavior, not on internals).
